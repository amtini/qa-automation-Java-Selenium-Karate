----------------------
ON THE SUITE'S EVOLUTION
----------------------

POMs are living artifacts. They have to evolve as the site changes:
when a new element appears, when the DOM structure is modified, when
new flows are added. What we deliver here is a snapshot of the current
state of the site (SauceDemo), modeling only what is needed to cover
the requested flow.

If the site changes, the Page Objects are the first place to touch.
The test stays untouched as long as the Page Objects' public interface
is preserved. That is the gain of the pattern.

-----------------------------------------
DESIGN DECISIONS MADE AND THEIR RATIONALE
-----------------------------------------

1. Page Object Model pattern with HeaderComponent
   - Each page of the site has its own POM class.
   - The header (cart icon + badge) appears on multiple pages, so it
     was modeled as a HeaderComponent to make it reusable instead of
     duplicating code in each Page Object.

2. Adding products through two different flows
   - One product is added from the inventory list.
   - The second product is added from the product detail page.
   - Reason: it satisfies the "add two products" requirement while
     covering two real user paths, not just one. This widens the
     possibility of using different ways to add to the cart in future
     tests.

3. Cross-view validation because there is no API or external source
   - SauceDemo has no public API, so I cannot compare the cart against
     a source of truth. What I do is capture name, price and
     description at the moment each product is added (from inventory or
     from the detail page) and compare that data against what the cart
     and step 2 show.
   - It detects any discrepancy between what the user adds and what the
     site ends up showing. If the inventory page already showed bad
     data from the start, that would require an external API to
     validate it, which is out of scope.

4. Subtotal validation at checkout
   - I compute the sum of the cart prices and compare it against the
     "Item Total" of step 2.
   - I do not validate the tax or the final total, for two reasons.
     One: I don't have access to the rate SauceDemo applies (it's not
     published as an API). Two: validating that "subtotal + tax =
     total" would be testing the site's math, not the purchase flow,
     which is what the exercise asks for.
   - If in the future I had the rate available, or wanted to validate
     the internal consistency of the calculation, it could be added as
     an improvement.

5. Buttons not exercised (Continue Shopping, Cancel)
   - The site has "Continue Shopping" buttons in the cart and "Cancel"
     buttons in the checkout steps. They were not modeled in the Page
     Objects because they are not part of the flow requested in the
     instructions.
   - Reason: it is preferable to avoid untested code.

6. Payment and shipping information in CheckoutStepTwo
   - The payment and shipping information is static; it has no
     interactive control or selector to choose other options. This
     means I can only validate the text "SauceCard #31337" and the
     "Free Pony Express Delivery!".
   - Why do I validate it? To lay the groundwork for future validation
     of the selected payment and shipping method at checkout. On a real
     site, these fields would be selectors and we would need different
     strategies to be able to complete the purchase.

7. Multi-browser support
   - The DriverFactory supports Chrome and Firefox through the
     -Dbrowser=<chrome|firefox> parameter. Default: Chrome.
   - Edge was left out because Microsoft retired the azureedge.net CDN
     where Selenium 4.16.1 looks for the msedgedriver, so the driver
     download fails with a DNS error. To add it, Selenium would have to
     be bumped to >= 4.21, which I did not do at the end of the project
     to avoid risking regressions in the Chrome flow. It remains a
     future improvement.
   - Opera and Brave officially lost support in Selenium 4, which is
     why they were also left out.
   - The DriverFactory pattern leaves the door open to add new browsers
     with minimal code.

8. Configurable headless mode
   - Having headless mode as a flag (-Dheadless=true|false) allows
     switching between seeing the browser during manual debugging and
     running without UI in a CI.
   - This also helps us configure a possible future CI such as Jenkins
     or GitHub Actions.

9. Centralizing configuration in Config
   - All config (browser, headless, baseUrl, waits) is read from a
     single class. No other file in the project touches
     System.getProperty or the .properties.
   - The config.properties file defines the default values; the -D
     flags override them at runtime. Useful so CI can use other values
     without editing anything.
   - The methods return the correct type (boolean, int, String), so the
     parsing is not repeated every time it is queried.
   - If the config source changes later (env vars, vault, etc.), only
     Config.java is touched.

10. Separation between data/ and factories/
    - data/ contains the "shape" of the data: records (Product,
      CheckoutFormData) and constants (Products). It defines WHAT a
      product is, not how to create it.
    - factories/ contains the "construction": DriverFactory creates
      WebDrivers, DataFactory creates CheckoutFormData with random
      values via DataFaker. It defines HOW an instance is produced.
    - The separation matters because the same record can be created
      from several places. The factory generates it with fake values,
      and in our case I also capture Products by reading the DOM,
      either from InventoryPage or from ProductDetailPage.

11. Locator strategy:
    - My experience with Playwright got me used to using data-testid on
      elements. A quick look at SauceDemo's HTML confirmed that all the
      important elements have their data-test attribute, so I started
      directly with that strategy.
    - Why did I choose to use only data-test?
      It is the best way to have an explicit contract between devs and
      QA. Any other locator (id, CSS class, XPath) can change with a UI
      refactor. data-test ones don't, because they exist specifically
      for automation.
    - I also completely avoided locating by visible text (By.linkText,
      etc.) in the main flow, because it breaks with any translation of
      the site.

12. Accessor pattern to expose components to the tests
    - I had two options for the tests to use HeaderComponent from each
      page: delegate each method (page.cartItemCount() internally calls
      header.cartItemCount()) or expose the component with an accessor
      (page.header().cartItemCount()).
    - I chose the accessor. If tomorrow I add logout() or
      resetAppState() to HeaderComponent, I don't have to touch the 5
      pages that use it.
    - The test stays explicit: "I query this page's header"; the
      composition is not hidden.

13. Custom reporting with PASSED / BUG / FAILED categories + exportable JSON
    - I started with Allure and ended up removing it. I prefer building
      my own output with categories by error type and an exportable
      JSON that can feed Jira, Linear or a Slack webhook. More portable,
      it doesn't depend on an external CLI, and the error categorization
      is explicit.
    - TestListener categorizes each failure by exception type:
      * AssertionError -> "BUG" (the site did not meet expectations).
      * Any other exception (Selenium, NPE, etc.) -> "FAILED"
        (automation or infra problem).
      * Test with no exception -> "PASSED".
    - Each result is printed to the console with a clear banner (status,
      test, time, reason, location, screenshot if applicable).
    - At the end of the suite, target/test-results.json is written with:
      * totals by category
      * a list of tests with name, status, duration, reason, location,
        and path to the screenshot if it failed.

14. Support for multiple environments
    - Config.java reads the -Denv variable (default "dev"). If I run
      mvn test -Denv=staging, baseUrl points to the staging environment
      instead of the default.
    - This way the same suite runs against different environments
      without touching the Page Objects. The alternative URLs are
      placeholders because SauceDemo only exposes the public site, but
      the pattern is ready for a private app with several deployments.


------------------------------------------
OBSERVED LIMITATIONS OF THE SITE (SauceDemo)
------------------------------------------

- There is no quantity selector. Once a product is added to the cart,
  the "Add to cart" button is replaced by "Remove", so you can only
  have one unit of each product.
- The payment and shipping data is static. You cannot simulate a card
  failure, an invalid address, etc.
- There is no real authentication; they are mock users. There is no
  persistent session or tokens.
- There is no search functionality or filters, only a sort dropdown.
- The product detail page is practically identical to the inventory
  item in terms of information; it only adds a larger image.

------------------------------
FUTURE IMPROVEMENTS FOR THE SUITE
------------------------------

Parameterized test (instead of many separate tests):
In previous projects I worked with a parameterized approach where a
single test method runs through N scenarios defined as data, instead of
duplicating the flow across multiple separate tests. For example, in a
previous checkout project I handled scenarios with dimensions like
locale, currency, payment method, product type, user type (guest vs
logged-in) and whether the flow is completed or cancelled mid-way. A
single .spec.ts iterated over the list and generated one test per
combination.

Applied to SauceDemo, this would allow covering from a single test
method:
- Different users (standard_user, locked_out_user, problem_user,
  performance_glitch_user) each with a different expected behavior.
- Different ways of completing the flow: complete everything, cancel at
  step 1, cancel at step 2, go back from the cart with "Continue
  Shopping".
- Different product combinations.
- Edge cases with expectedError (blocked login, missing form data,
  etc.).

The TestListener's categorization (PASSED / BUG / FAILED) pairs well
with this approach: each scenario produces its own entry in the JSON
with its status, without the report's test list inflating artificially.

Additional tests this parameterized approach would cover:
- Login with locked_out_user (error message validation).
- Login with problem_user (intentionally broken UI, useful for
  detecting known problems in components that render products).
- Login with performance_glitch_user (artificial delays, useful for
  validating that the explicit waits hold up).
- Validation of the "Continue Shopping" flow from the cart.
- Validation of the "Cancel" flow at each checkout step.
- Validation of the final total and tax at checkout step 2 (would
  require knowing the rate the site applies).

Infrastructure:
- CI/CD pipeline with GitHub Actions running headless so the suite is
  executed on every PR or push to main.
- Parallel execution of the suite when it grows, to keep feedback times
  low.

Code quality:
- Externalize the parameterized test data to a JSON or CSV file, so
  non-Java QAs can add scenarios without recompiling.

----------------------------------------------
THINGS THAT HAPPENED DURING DEVELOPMENT AND EXECUTION
----------------------------------------------

- During the first execution in Chrome, a native browser popup appeared
  warning that the password "secret_sauce" was in a data breach
  (Chrome's password leak detection). Since SauceDemo uses well-known
  public credentials, that browser check interrupts the test flow. I
  solved it by adding preferences to the DriverFactory to disable the
  password manager and breach detection (credentials_enable_service,
  profile.password_manager_enabled,
  profile.password_manager_leak_detection). Also applied to Firefox
  with its equivalent prefs.

- Selenium 4.16.1 emits a warning about Chrome DevTools Protocol (CDP)
  when the installed browser is a newer version than the one supported
  by the library. On my machine, Chrome 148 vs CDP supported up to
  v124. It does not block execution or affect functionality: CDP is
  used for advanced features (network interception, geolocation,
  full-page screenshots) that this test does not need.

- Execution time of the full flow in Chrome (non-headless, local):
  ~2.2 seconds for the test and ~5 seconds in total including the
  browser spin-up. Fast and consistent between runs.

- Test result in Chrome: PASSED. All assertions (cart count, items in
  cart, subtotal, items in step 2, payment info, shipping info,
  confirmation message) passed on the first successful execution.

- I tried Allure as a reporter but ended up removing it.
  The reason: for a single happy-path test, configuring and explaining
  Allure (including the external CLI) was overkill. It was replaced by
  custom reporting in TestListener that generates target/test-results.json
  with PASSED/BUG/FAILED categories, clearer and more portable.
