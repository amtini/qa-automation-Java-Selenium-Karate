----------------------
ON THE SUITE'S EVOLUTION
----------------------

Karate's feature files are living artifacts. They have to evolve as the
API changes: when an endpoint modifies its schema, its status code or
its behavior, the feature is the first place to touch. As long as the
API's contract is preserved, the test keeps passing.

-----------------------------------------
DESIGN DECISIONS MADE AND THEIR RATIONALE
-----------------------------------------

1. Karate as the framework
   - I used Karate because writing API tests in Gherkin reads more
     easily than in plain Java, and a non-Java QA can still maintain
     it.
   - It also comes with everything out of the box: HTTP client, JSON
     validation, path and param handling, rich HTML report. With
     RestAssured you would have to add more libraries to get to the
     same place.

2. A single Scenario with 4 chained steps
   - The brief talks about "the pet that was entered" and "the pet that
     was modified". The definite articles make it clear that it is the
     same pet throughout the whole flow.
   - So I built a single Scenario with the 4 chained steps (POST -> GET
     by ID -> PUT -> GET by status), sharing petId and name across
     operations. The alternative of 4 separate scenarios would need a
     setup POST in each one and more trips to the sandbox.

3. Random petId at runtime
   - In the Background I generate a random petId between 1,000,000 and
     2,000,000.
   - The PetStore sandbox is public and shared. If I hardcode an ID I
     could collide with another user's pet and the test breaks without
     there being a real bug.

4. Variables centralized in the Background
   - I declare petId, originalName and updatedName only once in the
     Background and reuse them across the 4 steps.
   - The feature stays cleaner and you can see at a glance that it is
     the same resource from end to end.

5. karate-config.js as the single place for configuration
   - The baseUrl, the Pet factory and other helpers live in
     karate-config.js. The features don't hardcode URLs or objects.
   - If tomorrow the environment or the shape of the Pet changes, I
     touch that file and the features stay intact.

6. Three independent validations per step
   - In each of the 4 steps I validate three different things:
     * That the response matches the Pet schema (the shape).
     * That the Content-Type header is application/json.
     * That the specific data (id, name, status) is what I sent.
   - If the shape breaks, it's the API changing structure. If the
     header breaks, it's a change in the Content-Type. If the data
     breaks, it's a logic bug. Each failure points to a different
     place.

7. JUnit 5 as the runner via PetRunner.java
   - Maven runs .java classes, not .feature files directly. So I built
     PetRunner.java as a minimal 4-line wrapper that JUnit detects and
     that calls Karate.run("pet").
   - All the test logic is in the .feature; the runner is just a
     bridge.

8. Reporting
   - For the API I use the reports Karate generates on its own in
     target/karate-reports/ (HTML and JSON).
   - For the E2E I built a TestListener that writes test-results.json
     with passed / bugs / failed / skipped.
   - The integrations with Slack, Jira or whatever I designed outside
     of the test code: a script reads the JSON and triggers the action.
   - Future improvement: unify both JSONs into a single one.

9. Support for multiple environments
   - karate-config.js reads the karate.env variable (default "dev"). If
     I run mvn test -Dkarate.env=staging, baseUrl points to the staging
     environment instead of the default.
   - This way the same suite runs against different environments (dev,
     staging, local mock, prod) without touching the features. The
     alternative URLs are placeholders because PetStore only exposes
     the public sandbox, but the pattern is ready for a private API
     with several deployments.

10. Reusable schema to validate the response's shape
    - In petContract.feature I defined the expected shape of the Pet
      (petShape: types of each field and valid enums). In each step of
      the feature I match against that schema.
    - If the API removes a field or changes a type, the test breaks
      right there without having to review which fields I checked
      where. I use match contains and not match == so that if they add
      new fields the test keeps passing.

11. defaultPet factory in karate-config.js
    - The common Pet body (category, photoUrls, tags) I built as a
      function defaultPet(petId, name, status) in karate-config.js. The
      POST and the PUT call it varying only those 3 parameters.
    - If tomorrow the request structure changes, I modify the factory
      and the two endpoints get updated together. It avoids duplicating
      the object and makes the feature more readable.

12. Response time check
    - In each helper I added an assert responseTime < 3000 (3 seconds)
      after the status check.
    - It's not a performance test, but if an endpoint starts taking
      longer than that, there's already something odd to look at before
      monitoring finds out. For PetStore I used 3 seconds because it's a
      public sandbox and times vary. On a production API I would tighten
      the number.

13. Content-Type validation
    - In each helper I validate that the response's Content-Type is
      application/json.
    - Sometimes bugs show up in the headers (charset set wrong,
      content-type that changes after a proxy). Validating the header
      gives me the guarantee that I parse JSON because the API declares
      it as JSON, not by chance.

------------------------------------------
OBSERVED LIMITATIONS OF THE API (PetStore)
------------------------------------------

PetStore is a public demo sandbox. That brings things that would not be
acceptable in a real API, and that condition what can be tested:

- Shared data: anyone can create, modify or delete resources. My pet
  may not survive if another user deletes it. That's why I use a large
  random petId, to minimize collisions.

- Sometimes findByStatus takes a moment to reflect the PUT's changes.
  If the test runs at exactly that moment, the filter may not find the
  just-updated pet.

- There are no idempotency keys, transactionId or createdAt in the
  responses. In a payments or banking API those patterns are
  indispensable; here they can't be demonstrated.

- The endpoints are completely open, with no real authentication. In a
  serious system you would have to test login with tokens (JWT, OAuth2,
  refresh). Here it doesn't apply.

------------------------------
FUTURE IMPROVEMENTS FOR THE SUITE
------------------------------

Tests that could be added without touching the main flow:

- Negative cases: GET with a non-existent ID -> 404, POST with an
  invalid body -> 4xx, PUT with missing required fields, etc.

- Concurrency: simultaneous PUTs over the same resource to validate
  locking (ETag / If-Match) if the API supports it.

Infrastructure:

- CI/CD with GitHub Actions running the suite on every PR.
- Local mock with WireMock so as not to depend on the sandbox's uptime.
- Basic performance tests with Karate Gatling.

Things that would apply in a banking or production API (can't be shown
here):

- Idempotency keys (same key + body -> same response).
- Full auth flow: login -> token -> refresh -> logout.
- Audit trail in responses (createdAt, transactionId).
- Rate limiting (X-RateLimit-Remaining headers, 429).
- HTTPS, ciphers, cert pinning.

----------------------------------------------
THINGS THAT HAPPENED DURING DEVELOPMENT AND EXECUTION
----------------------------------------------

- Karate generates a rich HTML report
  (karate-reports/karate-summary.html) without configuring anything. It
  brings a scenario tree, full request and response of each step,
  timings. Useful for manual inspection.

- The full feature runs in ~2.7 seconds for the 4 chained operations.
  Reasonable for a public sandbox.

- In all my runs, findByStatus responded fine and the just-updated pet
  appeared in the array. The eventual consistency I mention in
  Limitations was never triggered; it remains a documented possibility
  and not something I saw live.

- Something that happened to me with contract testing on public
  sandboxes: I tried applying match each contains petShape over ALL the
  pets returned by findByStatus and the test failed, because other
  users created pets with dirty data (some without category, others
  with junk names). The lesson: the schema only applies to your own
  resources, not to third-party data. The solution was to filter the
  array by petId and only then validate the schema over my pet.

- To integrate Karate with Maven Surefire I built a minimal JUnit 5
  runner (PetRunner.java) and configured testResources in the pom.xml
  so the .feature located in src/test/java is copied to the test
  classpath.

- The 4 steps of the scenario passed on the first execution.
